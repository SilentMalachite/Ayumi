# 貢献ガイド

歩み（Ayumi）への貢献をありがとうございます。小規模な福祉事業所で使う記録アプリのため、
**明快さと、小さくレビューしやすい変更** を何よりも重視します。

## はじめに

開発環境のセットアップは [README](README.md#セットアップ) を参照してください。

```bash
mix setup        # 依存取得・DB作成・初期データ・アセット
mix phx.server   # 起動
mix review       # 品質ゲート（後述）
```

## 開発の進め方

- **結論から。** まず答え・判断を述べ、その後に理由を続けます。
- **テストファースト（TDD）。** 実装より先に失敗するテストを書きます。まずはコンテキスト関数と
  Ecto changeset から。
- **最小の差分。** 目的を果たす最小の変更にとどめ、変更したファイルと関数・理由を明示します。
- **一度に一つ。** 無関係な変更を混ぜません。気づいた別の改善点は列挙して相談し、同じ変更には
  含めません。
- **設計してからコード。** 非自明な変更は、まず構造とデータの流れを説明し、合意してから実装します。
- 大きなリファクタリング、合意済みモデルを超えるスキーマ変更、依存追加は、着手前に相談してください。

## コード規約

- ドメインロジックは Phoenix **コンテキスト** に置きます。LiveView は薄く保ち（assign と
  イベントハンドラのみ）、ロジックはコンテキストに委譲します。
- バリデーションはすべて Ecto **changeset** に置きます。LiveView 内でその場限りの検証をしません。
- 識別子・モジュール名・コメントは **英語**。利用者向け文字列は **日本語**（スタッフは日本語話者）。
  日本語文字列は `gettext` か1か所にまとめ、コード中に散らばらせません。
- 状態を導出するもの（例:「イベント群から現在の段階を求める」）は **純粋関数** にして、
  単体テストしやすく保ちます。
- **追記専用の原則を壊さない。** 状態変更は既存行の上書きではなく新しい行で記録します。訂正も
  新しい行です。「現在の状態」は保存せず最新行から導出します（`current_stage` のような可変カラムを
  足さない）。

設計の背景は [`CLAUDE.md`](CLAUDE.md) に詳しくあります。

## テストと品質ゲート

- テストは ExUnit。コンテキストと changeset を直接テストし、主要な操作フローには LiveView テストを
  足します。
- 作業は次が通って初めて「完了」です。

```bash
mix review   # format確認 → 警告をエラー扱いでコンパイル → credo → test
```

## ブランチとコミット

- `main` から作業ブランチを切ります。`main` へ直接コミットしません。
- コミットメッセージは **Conventional Commits** に従います。

```
<type>: <要約>

<本文（任意）>
```

`type` は次のいずれか: `feat` / `fix` / `refactor` / `docs` / `test` / `chore` / `perf` / `ci`

例: `feat: 短期目標の進捗記録を追加`

## プルリクエスト

- コミット履歴全体（最新コミットだけでなく）を踏まえた要約を書きます。
- 変更点・背景・**テスト計画** を記載します（[PR テンプレート](.github/PULL_REQUEST_TEMPLATE.md) 参照）。
- `mix review` が通っていることを確認してから提出します。
